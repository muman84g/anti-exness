Use this only when the local project tree is not suitable for the requested push. Keep clones outside `C:\botter` and `C:\botter\bot`:

```powershell
$STAMP = Get-Date -Format yyyyMMdd_HHmmss
$CLONE_ROOT = 'C:\Users\muuma\Downloads\codex-temp'
New-Item -ItemType Directory -Force -Path $CLONE_ROOT | Out-Null
$CLONE = Join-Path $CLONE_ROOT "anti-exness-push-$BOT-$STAMP"
```

Copy the contents of the target folder, not the folder onto itself:

```powershell
$SRC_ROOT = 'C:\botter\bot'
$SRC_BOT = Join-Path $SRC_ROOT $BOT
$DST_BOT = Join-Path $CLONE $BOT
Copy-Item -LiteralPath "$SRC_ROOT\BOT_BACKTEST_MAP_ja.md" -Destination "$CLONE\BOT_BACKTEST_MAP_ja.md" -Force
Get-ChildItem -LiteralPath $SRC_ROOT -Filter 'github*push.txt' | Select-Object -First 1 | Copy-Item -Destination $CLONE -Force
New-Item -ItemType Directory -Force -Path $DST_BOT | Out-Null
Get-ChildItem -LiteralPath $SRC_BOT -Force |
  Where-Object { $_.Name -ne 'live_config.py' } |
  Copy-Item -Destination $DST_BOT -Recurse -Force
Get-ChildItem -LiteralPath "$DST_BOT\__pycache__" -Force -ErrorAction SilentlyContinue |
  Where-Object { $_.Name -like 'live_config*.pyc' } |
  Remove-Item -Force
```

If compose changes are authorized, add or update only `$SERVICE`. Preserve unrelated service blocks and mounts.

## CentOS Follow-Up

After a successful push, report these target-specific next steps; do not run them unless separately authorized on that host:

```bash
cd /home/muu/python_program/anti-exness

```

For this CentOS Docker deployment, treat `git pull` as host-file update only.
If live-bot source, params, artifacts, compose mounts, or helper files changed,
the running container should be considered stale until the target service is
recreated. Do not say the running bot has the update just because `git pull`
finished.

For a first deployment, place the local sensitive `botNN/live_config.py` on CentOS manually before any start. Do not commit it.

No-order self-test, after replacing `<NN>` and confirming the runner path:

```bash
sudo docker compose run --rm --no-deps exness-bot-<NN> python3 /app/bot<NN>/live_s<NN>_bot.py --self-test
```

After the no-order self-test passes, recreate only the authorized target to
make the updated files active in the running container:

```bash
sudo docker compose up -d --no-deps --force-recreate exness-bot-<NN>
sudo docker ps --filter name=exness-bot-<NN>
sudo docker compose logs --tail=80 exness-bot-<NN>
tail -n 80 bot<NN>/logs/s<NN>_bot.log
```

Before a first shadow-forward start, confirm the target params keep real trading disabled and shadow mode enabled unless the user explicitly authorized another mode.
